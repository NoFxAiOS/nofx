# NOFX 架构/开发决策记录

## 2026-03-21 — 决定先做项目接管与文档治理，而非直接功能开发

### 背景
用户目标是推进加密货币 AI 交易系统的深度开发优化，并特别强调：
- 熟悉完整架构
- 读懂各模块
- 形成中文结构文档与注释
- 优化记忆与 token 使用
- 保证后续新增功能有良好衔接

### 决策
先完成“认知地基 + 治理文件 + 测试基线”，再进入正式功能开发。

### 原因
1. 交易系统属于高风险系统，直接改功能容易引入不可见错误
2. 没有统一架构文档，后续会话衔接成本高
3. 收益优化必须建立在稳定性和一致性之上
4. 文档化与决策记录可以显著降低 token 消耗和上下文丢失风险

### 影响
- 短期内进度看起来偏“治理型”而非“功能型”
- 中期会明显提高后续开发速度与质量
- 为多 agent / 多子任务协作提供统一基础

## 2026-03-23 — 决定先完成阶段性代码交付，再进入接管工程收口

### 背景
在首轮推进中，已经完成：
- 后端测试基线通过
- 前端测试/构建基线通过
- 首轮外部问题清理
- 首轮低风险性能优化
- 首轮前端 API 收束

但用户明确指出：
- 不能把“代码可交付”误当作“接管任务整体结项”
- 中文文档、中文注释、项目记忆归档、流程固化、本地记忆固化都属于原始任务的一部分

### 决策
将当前项目状态明确拆成两层：
1. **版本代码交付层**：测试绿、构建绿、外部阻塞项清理后，可阶段性交付
2. **接管工程层**：必须继续补齐结项总文档、记忆归档总表、关键入口中文注释、可信性/风险边界说明，才能宣布整体阶段性结项

### 原因
1. 交易系统的“代码能跑”不等于“系统已可信收口”
2. 没有项目级总归档，下次对话和后续二开仍会丢失上下文
3. 没有总控结项文档，就无法明确哪些已完成、哪些未完成、下一阶段如何承接
4. 中文化如果只停留在文档层，后续直接读代码仍然成本很高

### 影响
- 当前立即进入“结项资产落仓”阶段
- 后续优先补总控文档、项目记忆总表、关键入口中文注释
- 二次开发前，项目会拥有更强的仓库内记忆与承接能力
