# 持续推进工作流（TaskFlow × Agentic Coding）

> 目的：解决长期任务在模型波动、上下文漂移、子任务超时、会话中断后容易半途断开的执行问题。
> 适用范围：`nofxmax` 的长期接管、缺陷定位、实盘问题排查、跨多轮验证的交付任务。

---

## 1. 为什么需要这份流程

当前项目不是一次性 bugfix，而是持续接管中的 AI Trading OS / 自动交易平台。仅靠单轮对话推进，会出现这些典型失效模式：

1. **模型波动**：超时、切模、回复质量波动，导致中途停摆。
2. **上下文漂移**：会话变长后，任务真实状态只存在聊天上下文里，容易丢。
3. **子任务孤岛化**：排查、测试、修复、文档、验证分散进行，没有统一 owner 状态。
4. **完成标准不稳**：没有先写 acceptance，容易陷入“改了一点，但到底算不算完成” 的循环。

因此，后续复杂任务不再只依赖主会话即时记忆，而采用：

- **TaskFlow 思想**：把任务当作可恢复、可继续、可等待、可收口的 flow。
- **Agentic Coding 思想**：每次改动先锁 contract，再做最小变更，再给出证据。

这不是为了“形式化”，而是为了让任务在中断后还能继续，而不是重新摸索。

---

## 2. 组合原则

### 2.1 TaskFlow 负责什么

用于管理长期任务的“持续推进能力”：

- 给每个任务一个稳定身份（flow）
- 明确当前步骤（current step）
- 记录最小可恢复状态（state）
- 标记等待原因（wait reason）
- 关联子任务 / detached runs
- 在中断后能继续，而不是重新开题

### 2.2 Agentic Coding 负责什么

用于管理每次具体实现的“交付质量”：

- 先锁定目标与 acceptance
- 用最小 diff 修问题
- 先证明失败，再证明修复
- 给出 handoff 级别的验证结果与风险说明

### 2.3 二者如何配合

- **TaskFlow** 管“这件事怎么持续推进到底”
- **Agentic Coding** 管“每一步改动怎么做得稳、准、可回滚”

一句话：

> TaskFlow 负责不断线，Agentic Coding 负责不乱改。

## 3. 标准执行模板

后续涉及多步骤、跨验证、可能中断的任务，统一按以下模板推进。

### Step 0：建立 Flow 卡片

在开始做事前，先写清这 5 件事：

- **Objective**：这次到底要交付什么
- **Acceptance**：怎么证明完成
- **Non-goals**：这次明确不动什么
- **Current Step**：现在在第几步
- **State**：恢复时必须知道的最小事实

推荐最小模板：

```md
## Flow Card
- Objective:
- Acceptance:
- Non-goals:
- Current Step:
- State:
- Waiting On:
```

### Step 1：先做定位，不急着改

必须先回答：

- 问题能否稳定复现？
- 数据证据在哪里？
- 测试覆盖到了哪一层？
- 真实环境与测试环境差异是什么？

若这些问题还没回答清楚，不进入大改。

### Step 2：锁定 Contract

每个实现动作都要在动手前写清：

- 只改哪条链路
- 为什么这条链路是当前最高优先级
- 风险边界在哪里
- 验证命令是什么

推荐模板：

```md
## Contract
- Objective: 
- Acceptance:
  - 
  - 
- Non-goals:
  - 
- Constraints:
  - 最小改动
  - 不扩大 blast radius
```

### Step 3：做最小 Change Set

执行要求：

- 优先做**最小闭环修复**
- 不把多个历史问题混成一次提交
- 不顺手带上无关重构
- 能在局部收束，就不做全局大改

### Step 4：Trace and Test

至少提供两类证据：

1. **前/后对比证据**
   - 修前失败的现象 / 测试 / raw 数据
   - 修后通过的结果 / 日志 / 测试

2. **交付证据**
   - `go test ./...`
   - `cd web && npm test`
   - `cd web && npm run build`
   - 必要时补充定向测试

### Step 5：更新 Flow 状态

每完成一个阶段，都要写回：

- 当前做到哪一步
- 还剩什么
- 卡在哪里
- 下次从哪里接续

禁止把“续作点”只留在聊天里。

### Step 6：文档与记忆固化

每次阶段性完成后，最少更新其中之一：

- `docs/DEVLOG.md`
- `docs/TODO.md`
- `docs/PROJECT_MEMORY_ARCHIVE_CN.md`
- workspace 下 `memory/YYYY-MM-DD.md`
- 必要时 `MEMORY.md`

### Step 7：提交收口

通过验证后必须提交，提交信息应反映真实动作，例如：

- `docs: add durable execution workflow for long-running tasks`
- `fix: migrate legacy protection value shape on strategy update`
- `test: cover strategy payload protection mode persistence`

## 4. 适用于 nofxmax 的任务分层

### 4.1 主会话负责

主会话只负责：

- 定义目标
- 决定优先级
- 汇总阶段性结果
- 统一收口为 repo 记忆和提交

主会话不应该把所有细节都塞在一次长对话里硬顶。

### 4.2 子任务负责

适合拆出去的子任务：

- 真实链路比对
- 指定模块代码阅读
- 定向测试/构建
- 某个 API / 页面 / store 层证据采集
- 文档整理与 handoff 草稿

### 4.3 何时判定进入“等待态”

遇到以下情况，任务应进入等待态而不是假装还在推进：

- 等真实环境反馈
- 等外部系统返回数据
- 等用户做高风险决策
- 等某个 detached task 结果
- 等下一次可观测窗口（如实盘触发）

等待态必须写清：

- 在等什么
- 为什么不能继续
- 下次恢复入口是什么

## 5. 对当前项目的直接应用

### 5.1 保护系统问题排查

对 protection / execution 类问题，统一拆成四段：

1. **配置链定位**
   - 页面 payload
   - API update
   - store 持久化
   - runtime 读取源

2. **执行链定位**
   - 触发条件
   - 订单生成
   - 交易所参数适配
   - reconciler 生命周期

3. **验证链补强**
   - API 回归测试
   - 前端 payload 测试
   - runtime/订单层测试
   - 必要时 fixture + raw JSON 对照

4. **交付链收口**
   - DEVLOG
   - TODO
   - memory
   - commit

### 5.2 当前最适合按此流程推进的主线

当前优先主线：

1. Drawdown 多档委托生命周期
2. Break-even 实盘委托可观测性与成功率
3. Protection AI mode 真实环境差异定位

这些任务的共同点都是：

- 跨多层链路
- 不适合一次性拍脑袋修
- 需要证据与状态持续积累
- 容易因为会话切断而重来

因此必须按本流程推进。

## 6. 退出标准（Done Definition）

一个长期任务只有同时满足以下条件，才算真正完成：

1. 问题已被明确定位，且原因不是猜测
2. 修复已落代码，且 blast radius 可说明
3. 至少一条修前/修后证据链完整
4. 必要测试 / 构建通过
5. 仓库文档已更新
6. workspace 记忆已更新
7. 已提交 commit，可回滚、可追溯

如果只完成其中一部分，任务状态应明确标记为：

- running
- waiting
- blocked
- done

不要把“看起来差不多了”当作 done。

---

## 7. 执行约束（强制）

- 不再把长期任务仅依赖于单轮聊天上下文
- 不再在没有 acceptance 的情况下直接大改
- 不再让“下一步做什么”只存在脑内或会话里
- 不再把中断后的重新摸索当作正常成本


## 8. 编程任务自治升级（2026-04-17）

针对“拿到任务后反复回来问、推进不够连贯”的问题，后续 coding 类任务默认执行以下规则：

1. **先做再问**：能通过读代码、跑测试、起 subagent、查文档解决的问题，不先回来打断用户。
2. **只在这些情况再问**：
   - 高风险外部动作
   - 不可逆操作
   - 明确产品方向分叉
   - 需要用户提供机器外信息/凭据/人工决策
3. **默认使用 durable workflow**：Flow 卡片 → Contract → 最小改动 → 证据验证 → 文档记忆 → 提交。
4. **默认使用 subagent 分担**：只读调查、长耗时分析、测试与验证任务，优先交给 subagent 并等待回传，不在主会话硬顶。
5. **阶段汇报替代中途询问**：以“已完成 / 风险 / 下一步”形式汇报，而不是每走一步都索取确认。

目标不是少沟通，而是减少低价值打断，提高连续交付能力。
