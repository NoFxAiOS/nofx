> 目的：把项目开始前已经约定的原则、分支策略、交付门禁、风险边界，固化为一份可执行工作流。
> 适用范围：当前 `nofxmax` 接管与持续交付阶段。
> 当前工作分支：`fox/project-takeover-baseline`

---

## 0. 总原则（必须优先服从）

### 0.1 优先级原则
按项目接管前约定：

1. **收益重要，但稳定性优先落地**
2. **没有正确性基线，就不做收益判断**
3. **不做高爆炸半径重构，优先做低风险可验证改动**
4. **每一步都必须能回滚、能验证、能说明**

### 0.2 执行原则
- 一次只推进一个明确目标
- 优先处理“外部问题 / 一致性问题 / 可交付阻塞项”
- 默认先修接口失配、死代码、注释误导、构建/测试问题
- 性能优化只做**不牺牲功能和稳定性**的改动
- 不用口头记忆规则，规则必须落在仓库文档中

### 0.3 风险控制原则
- **禁止**为了追求速度做大范围联动式重构
- **禁止**在未验证前把不稳定代码留在工作树
- 如果某文件存在历史残留/拼接污染，优先：
  1. 恢复到已验证稳定版本
  2. 再按更小颗粒重做
- 如果改动失败，第一目标不是“修到看起来差不多”，而是**恢复稳定基线**

## 2. 单任务执行标准（伏羲必须逐条执行）

每个任务都必须遵守以下步骤：

### Step 1：定义目标
必须能回答：
- 这次只改什么？
- 为什么先改它？
- 风险在哪里？
- 如何验证成功？

### Step 2：限制改动范围
- 一个任务尽量只动 1 个文件；必要时最多 2~4 个强相关文件
- 不顺手做“看起来也该一起改”的大杂烩式修改

### Step 3：先做最小改动
- 优先替换边界层代码
- 不改变业务语义
- 不改变已验证行为

### Step 4：立刻验证
前端改动必须至少执行：
- `cd web && npm test`
- `cd web && npm run build`

后端改动必须至少执行：
- `go test ./...`

### Step 5：失败处理
若验证失败：
1. 优先定位是否为残留代码污染 / 非本次逻辑问题
2. 若修复成本低，立即修
3. 若文件明显不干净或联动风险变高：**立即恢复稳定版本**
4. 绝不把红灯状态留在工作树

### Step 6：提交
- 改动通过验证后必须提交
- 提交信息必须反映真实动作，例如：
  - `fix:` 真实修复
  - `perf:` 性能优化
  - `refactor:` 边界收束/重构但不改语义
  - `docs:` 文档与流程固化

## 3. Git / 分支 / 提交流程

根据项目既定规则，当前采用开源侧 GitHub Flow 思路：

### 分支原则
- `main`：稳定分支
- `dev`：主开发基线
- 当前接管工作分支：`fox/project-takeover-baseline`

### 当前执行方式
在当前接管阶段：
- 所有接管、修复、优化、文档沉淀，都先落在 `fox/project-takeover-baseline`
- 待形成阶段性交付结果后，再整理合并策略

### 提交流程
1. 在当前工作分支完成任务
2. 每个通过验证的任务独立提交
3. 保持提交粒度清晰、可回滚
4. 提交完成后再次确认工作树干净

## 7. 对话执行规则（面向后续所有回合）

从现在开始，默认执行模式如下：

1. **不等待反复催促**，主动把一个任务闭环做完
2. **不做大而散的改动**，每轮只做一个安全可验证目标
3. **不把失败中间态当结果汇报**
4. **一旦完成验证，再汇报“已可交付状态”**
5. 如果用户说“继续”，默认按本工作流向下推进，不需要再反复询问是否继续
