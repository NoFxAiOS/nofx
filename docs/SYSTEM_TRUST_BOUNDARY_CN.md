# NOFX 系统可信边界与风险边界收口说明（中文）

> 状态：接管收口版 v1  
> 时间：2026-03-24  
> 适用分支：`fox/project-takeover-baseline`

---

## 1. 这份文档解决什么问题

这不是一份“系统能做什么”的宣传文档，而是一份：

- **系统在哪些边界内可信**
- **哪些地方目前只能认为是阶段性可信**
- **哪些地方仍然需要人工审计 / 后续专项验证**

对交易系统来说，这比功能列表更重要。

---

## 2. 当前可信边界的分层结论

## 2.1 基础运行层：相对可信

包括：

- 主程序启动链
- 配置加载
- 加密服务初始化
- 数据库初始化
- TraderManager 装载 Trader
- API Server 启动
- Telegram 可选启动
- 安全停机流程

这部分的可信度较高，原因：

- 启动链清晰且集中在 `main.go`
- 失败点大多采用 fail-fast
- 初始化顺序总体合理：配置 → 加密 → 存储 → 管理器 → API / Bot

### 当前结论
**可以认为“基础运行层已达到工程上可接受的稳定可信”。**

---

## 2.2 前后端管理面：相对可信

包括：

- AI 模型配置
- 交易所账户配置
- 策略配置
- Trader 配置
- 前端 Strategy Studio / Dashboard / Market / Settings 等页面
- API 与前端字段主链

本轮接管已完成：

- 首轮 API 收束
- 多处死代码 / 残留接口清理
- strategy protection 配置前后端首轮打通

### 当前结论
**配置管理面已达到“可交付、可继续迭代”的可信水平。**

但注意：

- 这不等于所有页面字段都经过逐项穷尽核验
- 仍可能存在局部历史残留接口未完全统一

---

## 2.3 AI 决策层：阶段性可信，不可视为绝对正确

包括：

- `kernel` prompt 构建
- AI 模型调用
- AI 返回解析
- AI 决策排序与执行触发

### 已有正向因素

- 决策链主路径清晰：上下文构建 → prompt → AI 返回 → 决策记录 → 执行
- 有决策日志、prompt、raw response、CoT trace 等落库/记录能力
- 有 safe mode 保护：连续 AI 失败后阻止继续开新仓

### 仍未完全可信的点

1. **AI 本身不是确定性组件**  
   即使提示词、上下文相同，也不能把输出视为严格确定。

2. **决策正确性不等于执行正确性**  
   AI 给出一个“看起来合理”的决策，不代表一定满足交易所实际约束或系统统计口径。

3. **策略 prompt 与风险约束混合存在**  
   一部分约束在 prompt 中，一部分在代码中，必须始终以代码强制规则为最终边界。

### 当前结论
**AI 决策层只能认为“可运行、可审计、有限保护下可用”，不能宣称为完全可信自动决策系统。**

---

## 2.4 交易执行层：可运行，但仍需交易所级一致性专项

包括：

- `trader/auto_trader*.go`
- 各交易所适配器
- 开仓 / 平仓 / 止盈止损 / 持仓同步 / 订单同步

### 已完成的加强

- 已有多交易所适配器统一接口
- 已接入 Full TP/SL Phase 1：
  - protection 配置结构
  - capability 骨架
  - manual protection plan
  - 开仓后挂保护单
  - 最小校验
  - 失败立即平仓，避免裸仓残留

### 仍未完全可信的点

1. **不同交易所 open order / stop / tp 语义并不完全一致**
2. **订单校验目前是“最小闭环”，不是“完备闭环”**
3. **改单 / partial / OCO / algo order 行为跨交易所差异仍大**
4. **异常恢复、重复下单、网络闪断后的最终一致性仍需专项核验**

### 当前结论
**执行层已经从“可下单”推进到“有保护意识的可执行”，但还没到“强一致高置信执行系统”的级别。**

---

## 2.5 风控层：已有代码强制边界，但仍未整体结项

当前已存在并可确认的风控边界包括：

- 最大持仓数限制
- 最小开仓金额限制
- 单仓名义价值比例限制
- 保证金使用率限制（部分路径）
- Safe mode（连续 AI 故障保护）
- Drawdown monitor（已有历史逻辑）
- Full TP/SL protection Phase 1

### 仍未完全可信的点

1. **风控规则分散**：存在于 strategy config、kernel、auto trader、exchange adapter 多处
2. **有些规则是代码强制，有些规则只是 AI 提示**
3. **Drawdown / Break-even / Ladder 等仍未全部统一进入 protection 体系**

### 当前结论
**风控层已经有可观的底线保护，但还没有完成“统一风控系统收口”。**

---

## 2.6 统计 / PnL / 展示层：可用但需口径专项

包括：

- 持仓历史
- 已平仓收益
- Competition 排行
- 前端收益展示
- equity snapshot

### 当前风险

- 不同交易所返回字段语义差异
- 本地重建与交易所回传之间可能存在偏差
- 手续费、部分平仓、外部平仓等场景仍需专项核对

### 当前结论
**展示层可作为运营/观察面使用，但“收益口径最终可信性”仍需专项核验后才能作为严格结算依据。**

---

## 3. 四条核心链路的当前可信度

## 3.1 启动链
结论：**高于其他链路，当前最可信。**

原因：

- 链路短
- 初始化顺序明确
- fail-fast 多
- 依赖关系相对单纯

## 3.2 决策链
结论：**可追踪、可审计，但不应过度信任 AI 输出本身。**

## 3.3 交易链
结论：**可执行，但交易所差异和异常一致性仍是核心风险来源。**

## 3.4 风控链
结论：**底线保护已有，但体系化统一仍未结项。**

---

## 4. 当前明确“不应误判为已完成”的事项

以下内容**不能**因为当前 test/build 通过就被视为完全完成：

1. 交易系统最终正确性
2. 跨交易所执行一致性
3. 收益统计最终口径一致性
4. 复杂保护单生命周期完整性
5. 极端异常场景恢复能力

---

## 5. 当前可对外承诺的级别

更准确的表述应是：

- 该系统已具备阶段性可交付能力
- 已完成接管期的基础治理、文档、首轮优化与 protection Phase 1 打底
- 适合继续进入第二阶段系统化增强
- 但不应被包装成“已完全验收的自动交易黑盒系统”

---

## 6. 建议的后续专项顺序

1. **执行一致性专项**  
   按交易所核验 open/close/stop/tp/cancel/amend 行为

2. **统计口径专项**  
   核对 realized pnl / fee / equity / leaderboard

3. **protection Phase 2**  
   Ladder / Drawdown / Break-even 统一进入 planner + execution

4. **trusted boundary 持续收口**  
   将“代码强制规则 / AI 提示规则 / 交易所能力限制”三类边界彻底分层写清

---

## 7. 当前结论（接管口径）

NOFX 当前不是“未整理的实验脚本”，也还不是“可无保留信任的成熟自动交易系统”。

更准确地说，它现在处于：

**已经完成接管基线、具备阶段性交付能力、并已开始进入系统可信性收口阶段的 AI Trading OS。**
